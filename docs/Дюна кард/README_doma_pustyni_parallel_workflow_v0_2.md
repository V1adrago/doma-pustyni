# Дома Пустыни — README рабочего процесса разработки и анализа

Версия: v0.2  
Дата: 06.06.2026  
Назначение: зафиксировать параллельную схему работы: анализ и проектирование делается здесь, кодовая реализация — через Claude, документация обновляется параллельно с кодом. Версия v0.2 дополнительно фиксирует механику боевых отрядов с общим HP и микро-таргетингом моделек.

---

## 1. Главная идея процесса

Проект ведётся в два параллельных потока.

**Поток 1 — анализ, геймдизайн, баланс, документация, промпты.**  
Здесь формируются механики, правила боя, баланс, логика юнитов, структура документов и точные промпты для Claude.

**Поток 2 — кодовая разработка.**  
Claude получает готовый промпт и правит код проекта: JS, Three.js, UI, сервер, онлайн, сборку и README кода.

Цель процесса — не смешивать размышления и кодовую реализацию.  
Сначала фиксируем механику словами. Потом отдаём Claude точное техническое задание.

## 2. Роли

### 2.1. Роль ChatGPT в проекте

ChatGPT отвечает за:

- анализ текущих файлов проекта;
- разбор логики механик;
- проектирование боевых правил;
- баланс юнитов, башен, экономики и постановки;
- формирование геймдизайн-документов;
- подготовку коротких и точных промптов для Claude;
- проверку того, не конфликтует ли новая механика со старой;
- упаковку изменений в MD и Word-документы;
- формирование следующего этапа разработки.

ChatGPT не должен писать код вместо Claude, если задача именно для Claude Code.  
Здесь нужно готовить Claude к реализации так, чтобы он не тратил контекст на лишние рассуждения.

### 2.2. Роль Claude в проекте

Claude отвечает за:

- реализацию механики в коде;
- поиск нужных файлов;
- внесение правок в JS/CSS/HTML;
- сохранение совместимости с текущей архитектурой;
- запуск `npm run build`;
- исправление ошибок сборки;
- обновление README кода;
- обновление `memory/project_doma_pustyni.md`;
- краткую фиксацию того, что реально изменено.

Claude не должен сам менять геймдизайн, если это не указано в промпте.  
Если в коде есть конфликт с механикой, Claude должен описать проблему, а не придумывать новый баланс.

## 5. Стандартный цикл работы над механикой

### Шаг 1. Обсуждение механики

Сначала обсуждается логика без кода.

Пример:

- как ставятся юниты;
- можно ли ставить за спину;
- где проходит граница поля;
- как работает поворот;
- кто должен иметь задержку;
- какие юниты являются исключениями.

На этом этапе не пишется промпт, пока механика не стала понятной.

### Шаг 2. Фиксация решения

После обсуждения формируется короткий вывод.

Пример:

- обычные боевые юниты ставятся на своей половине до середины;
- чужая половина закрыта;
- ставить за спину врагу можно, если точка находится на своей половине;
- враг должен развернуться и потерять время перед атакой;
- инженер, дрон и башнелом могут иметь ограничения постановки.

### Шаг 3. Обновление документов

После фиксации решения обновляются:

- общий MD проекта;
- отдельный MD по юнитам, если механика касается юнитов;
- Word-версии, если нужны для чтения и передачи;
- отдельный промпт для Claude.

### Шаг 4. Claude реализует код

Claude получает промпт и правит проект.

Обязательные действия Claude:

1. внести кодовые изменения;
2. запустить `npm run build`;
3. исправить ошибки;
4. обновить README.md;
5. обновить `memory/project_doma_pustyni.md`;
6. коротко написать, что реализовано.

### Шаг 5. Возврат результата на анализ

После работы Claude пользователь возвращает сюда:

- обновлённый README.md;
- STATUS.md, если есть;
- ошибку сборки, если есть;
- фрагменты кода, если механика работает неправильно;
- скриншоты или видео, если проблема визуальная.

Здесь проводится анализ результата и готовится следующий промпт.

## 9. Текущая зафиксированная механика постановки юнитов

Базовое правило:

**Обычные боевые юниты можно ставить на своей половине поля до середины.**

Запрещено:

- ставить обычных юнитов на чужой половине;
- ставить обычных юнитов около башен соперника;
- ставить обычных юнитов за пределами карты.

Разрешено:

- ставить перед врагом;
- ставить сбоку от врага;
- ставить за спину врагу, если точка находится на своей половине;
- использовать постановку для отвлечения врага от башни.

Ключевая боевая логика:

Если враг видит новую цель позади или сбоку:

1. он выбирает цель;
2. останавливается;
3. разворачивается;
4. тратит задержку на поворот;
5. только потом атакует.

Разворот должен быть частью баланса, а не просто анимацией.

## 10. Текущая классификация постановки юнитов

### Обычная постановка до середины

Эти юниты можно ставить на своей половине до центра:

- Ищейка Барханов / scout;
- Клинок Дома / swordsman;
- Пикейщик Каравана / spearman;
- Латник Пустыни / heavy;
- Гвардеец Чести / guard;
- Песчаный Стрелок / archer — допускается к тесту, но требует контроля баланса.

### Ограниченная постановка от тыла / башенной зоны

Эти юниты не должны стартовать у середины:

- Инженер Узла / engineer;
- Башнелом / assault;
- Дюнный Сокол / drone.

Причины:

- инженер должен пройти путь до центрального узла;
- башнелом слишком быстро создаёт угрозу башне;
- дрон не блокируется наземными ближними юнитами.

## 11. Текущая механика боевых отрядов с общим HP

Это новая целевая механика вместо прежней логики, где несколько моделек были только визуальным отображением одного юнита.

Главное правило:

**Одна карта = один Unit-отряд. HP общий. Но активные модельки внутри отряда реально участвуют в бою.**

У отряда остаётся:

- один общий HP;
- один общий HP-bar;
- один общий центр движения;
- одна основная коллизия;
- одна скорость;
- один deployPoint;
- одна онлайн-сущность;
- один общий приказ: идти по линии, атаковать врагов или давить башню.

У отдельных моделек внутри отряда появляется:

- свой визуальный слот в формации;
- статус активности от общего HP;
- свой локальный `subTarget`;
- свой вклад в общий урон;
- возможность временно сместиться к ближайшей цели в пределах радиуса связности.

Запрещено:

- делать отдельный HP для каждой модельки;
- превращать модельки в отдельные `Unit`;
- давать башням выбор конкретной модельки;
- синхронизировать каждую модельку по WebSocket;
- давать моделькам возможность убегать далеко от центра отряда.

`squadSize`:

| Юнит | squadSize | Боевая логика |
|---|---:|---|
| engineer | 1 | один инженер, без боевого разделения |
| scout | 2 | пара быстрых ищеек, две активные атаки при полном HP |
| swordsman | 3 | строй клинков, до трёх активных ударов |
| assault | 2 | ударная группа башнеломов, до двух ударов по башне |
| archer | 3 | группа стрелков, до трёх болтов |
| spearman | 3 | строй пикейщиков, до трёх уколов |
| drone | 1 | один воздушный дрон |
| heavy | 1 | один латник-танк |
| guard | 2 | два гвардейца со щитами |

Количество активных моделек зависит от процента общего HP.

Для `squadSize = 3`:

- 100–67% HP — 3 активные модельки;
- 66–34% HP — 2 активные модельки;
- 33–1% HP — 1 активная моделька;
- 0% HP — 0 моделек.

Для `squadSize = 2`:

- 100–51% HP — 2 активные модельки;
- 50–1% HP — 1 активная моделька;
- 0% HP — 0 моделек.

Боевой смысл:

- если все активные модельки бьют одну цель, их урон суммируется;
- если рядом несколько врагов, часть моделек может временно переключиться на ближайшие цели;
- если отряд теряет HP и модельки выбывают, суммарный урон падает;
- башня всегда наносит урон в общий HP отряда.

Пример для `swordsman`:

- базовый урон одной модельки: 45;
- при 3 активных модельках по одной цели: 45 + 45 + 45 = 135;
- если 2 модельки бьют цель A, а 1 моделька бьёт цель B: цель A получает 90, цель B получает 45;
- при падении HP ниже 66% остаются 2 модельки, максимум урона падает до 90;
- при падении HP ниже 33% остаётся 1 моделька, максимум урона падает до 45.

### 11.1. Радиус связности отряда

Модельки могут немного расходиться, но не могут разваливаться в разные стороны карты.

Базовые параметры MVP:

| Параметр | Значение |
|---|---:|
| squadLeashRadius для squadSize=3 | 1.15 |
| squadLeashRadius для squadSize=2 | 0.85 |
| squadLeashRadius для squadSize=1 | 0 |
| subTargetCheckInterval | 0.25 сек |
| minBenefitDistance | 0.45 |
| peelChance melee | 0.65 |
| peelChance ranged | 0.45 |
| peelChance одиночных юнитов | 0 |

Состояния модельки внутри отряда:

| State | Смысл |
|---|---|
| `formation` | моделька стоит на своём месте в строю |
| `peeling` | моделька временно смещается к ближайшему `subTarget` |
| `returning` | моделька возвращается в строй |

### 11.2. Правило выбора subTarget

Для каждой активной модельки:

1. Берётся мировая позиция модельки.
2. Ищутся враги в радиусе локального агро.
3. Если враг ближе к этой модельке, чем основная цель отряда минимум на `minBenefitDistance`, моделька может переключиться.
4. Переключение происходит не всегда, а через вероятность.
5. Если `subTarget` умер или вышел за `squadLeashRadius`, моделька возвращается в формацию.
6. Если моделька стала неактивной из-за потери HP, её `subTarget` очищается.

Важно:

`subTarget` не двигает весь отряд. Он влияет только на локальное смещение модельки и на цель её удара.

## 12. Как анализировать результаты Claude

После реализации нужно проверять не только сборку, но и соответствие механике.

Проверка:

1. Сборка проходит.
2. Код не сломал старые механики.
3. Новая механика работает в одиночном режиме.
4. Новая механика не ломает онлайн.
5. UI понятно показывает игроку, что можно делать.
6. Баланс не изменился случайно.
7. README.md обновлён.
8. memory/project_doma_pustyni.md обновлён.

Если что-то не работает, следующая задача формулируется не как «переделай всё», а как точечный фикс.

## 13. Принципы разработки проекта

1. Сначала простая механика, потом усложнение.
2. Не добавлять редкие исключения, пока базовый бой не работает.
3. Не менять баланс вместе с визуальной механикой, если задача не про баланс.
4. Не делать скрытые правила, которые игрок не видит.
5. Все зоны и ограничения должны подсвечиваться в UI.
6. Каждая новая механика должна иметь тестовые сценарии.
7. Каждая задача должна завершаться обновлением документации.
8. Кодовая реализация не должна сама менять геймдизайн.
9. Если механика спорная — сначала обсуждение, потом промпт.
10. Если Claude изменил архитектуру — это должно попасть в README.md.

## 14. Что делать при конфликте файлов

Если README.md говорит одно, а геймдизайн-документ другое:

1. считать README.md описанием текущего кода;
2. считать геймдизайн-документ целевым дизайном;
